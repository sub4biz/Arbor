# Quicker Start — lowering the time-to-first-"wow"

> Working plan for the `feat/quicker-start` branch.

## Problem

Arbor cannot run without an LLM API key. That is fundamental — it is an LLM
agent. But it means the **first thing a new visitor experiences is a wall**:
clone → `arbor` → "configure a provider first". 99% of GitHub visitors will
never paste a key, so they never see Arbor do anything. The repo's single best
asset — the hypothesis tree growing in real time — is invisible to almost
everyone who lands on it.

The fix is to separate two things that are currently fused:

- **"See it work"** — must be *zero* setup. No key, no model, no config.
- **"Use it for real"** — can have a setup step, but the barrier should drop
  from *pay* to *free signup*.

## Insight

The live dashboard is a **pure projection of the event bus**
(`cli/run_state.py`): every panel — idea tree, score chart, header counters,
reasoning feed — is a function of events. A finished run already leaves a
complete, append-only `events.jsonl`. So replaying a run is just: parse each
line back into an `Event` and re-emit it on a fresh bus wired to a fresh
dashboard, paced by the recorded timestamps.

That means **"watch a real Arbor run with zero setup" is one command away**, not
a big build — and the recording doubles as the shareable/viral artifact.

## Scope of this branch

### 1. Replay engine + `arbor replay` (this PR)
- `cli/replay.py` — load `events.jsonl`, drive the existing `RunDashboard` on a
  timeline (`--speed`, `--max-gap` to clamp idle gaps, read-only).
- `arbor replay <session-dir | events.jsonl>` — replay any real session.
- `arbor replay --demo` — replay a **bundled sample recording** with zero API,
  so a fresh clone can `arbor replay --demo` and immediately watch the tree grow.
- Demo asset shipped as package data (`cli/assets/demo_session/`), generated by a
  committed, reproducible script and clearly labelled as an illustrative sample
  (no benchmarked-result claims from synthetic scores).

### 1b. Shareable browser artifact — `arbor replay --html` (done)
- `cli/tree_export.py` + `cli/assets/tree_template.html` — a single
  self-contained HTML (vanilla JS + SVG, **zero external deps**) that animates the
  same recording: branches grow, pruned ones grey out, merged ones glow, with a
  scrub/play timeline. The page re-folds the events exactly as `run_state.py`
  does, so the browser tree is a provable projection of the same event stream
  (headless run of the template JS over the demo matches the terminal: 13.1×,
  4 merged, 2 pruned).
- This is the viral/shareable asset: drop the file into an issue, a slide, a
  tweet — every interesting run becomes content, generated by the product.

### 2. Free-tier quickstart (done)
- `cli/quickstart.py` — curated easy-start presets (Gemini / Groq free keys,
  Ollama local) mapping onto the `openai-chat` backend, plus a pure
  `build_llm_from_preset` for testability.
- `arbor setup` now leads with a "How do you want to connect a model?" menu —
  free/local first, "set it up myself" falls through to the full prompts.
- `arbor quickstart` — discoverable alias that lands on the same easy-start menu.

### 3. Real recorded demo (follow-up)
- Replace the synthetic sample with a recording of a real AlgoTune-knn run once a
  key is available, so `--demo` shows genuine agent behaviour.

## The onboarding ladder (target README first-screen)

```
1. arbor replay --demo   → 0 key, ~30s, watch a real tree grow   ← every visitor
2. arbor quickstart      → free key, ~2min, run the example       ← people who try
3. arbor (local Ollama)  → offline, 0 key                         ← privacy
4. arbor (paid API)      → real work                              ← serious users
```

Rung 1 is always zero. Today rung 1 *is* a paid API — that is the bug this
branch fixes.

## Design notes

- Replay re-emits only bus events (the dashboard's data source). `TreeSetMeta`
  metadata (baseline / metric direction) is **not** on the bus, so we read it
  from the session's `tree.json` and set the scalar header fields before
  replaying; the idea ledger stays empty so the tree builds up live.
- Gate events (`user.await` / `user.input_received`) are skipped during replay —
  they are interactive prompts, not research progress, and a dangling gate panel
  would stick at the end.
- Pacing: inter-event delay = `(ts[i] - ts[i-1]) / speed`, clamped to
  `[0, max_gap]` so long idle stretches (e.g. a 20-minute training run) don't
  make the replay boring.
- The recording's timestamps drive pacing only; emitted events reuse
  `bus.emit(type, data)` (handlers read `event.data`, never the timestamp).
